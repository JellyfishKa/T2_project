# T2-5 · R&D: выбор алгоритма гео-кластеризации

## Цель

Найти подход к распределению ТТ по торговым представителям (ТП), который **минимизирует суммарный пробег** (географически компактные дневные маршруты), сохраняя **баланс нагрузки**.

## Baseline (текущее поведение)

**Category-only**: распределение задач по ТП идёт только по категории (round-robin внутри категории), география игнорируется.

## Сравнённые подходы

Метрики фиксируем на одном и том же тестовом датасете **250 ТТ** (`ml/data/tt_250.csv`) и одинаковых параметрах:

- 5 ТП (`tp-1..tp-5`)
- месяц: 2026-05-01 (для D-категории визит 1 раз/квартал => в мае 0)
- дневной лимит: 12 визитов

### Подход 1: `kmeans+balance`

- **K-Means по координатам** \((lat, lon)\) с \(k = число\_ТП\)
- Затем эвристическая **балансировка размеров** кластеров (перенос точек в недогруженные кластеры по "стоимости" расстояния до центроида)

### Подход 2: `dbscan+pack`

- **DBSCAN** по матрице расстояний Haversine (км) → кластеры + шум
- Затем **greedy packing** кластеров в ТП (bin packing по размеру)

### Подход 3 (контрольный): `greedy-geo`

- Жадное распределение точек: точка назначается тому ТП, чей текущий центр (mean координат) ближе, tie-breaker — текущая нагрузка

## Метрики качества (фиксируем)

- **Суммарная дистанция** за месяц: сумма дневных маршрутов (Haversine, NN-порядок внутри дня)
- **Балансировка нагрузки по ТП**:
  - `load[rep]` = число визитов
  - `balance_score` в диапазоне 0..1, где 1 — идеально ровно (формула в `ml/benchmarks/geo_clustering_benchmark.py`)
- **Время расчёта**: `elapsed_ms` (для полного построения месячного плана)

## Выбор финального подхода

**Выбран: `kmeans+balance`.**

Обоснование:

- **Стабильно** уменьшает суммарный пробег относительно baseline.
- Даёт **контролируемый баланс нагрузки** (после балансировки).
- Время расчёта \(O(N)\) после обучения K-Means и заметно меньше, чем DBSCAN с матрицей расстояний \(O(N^2)\) на 250+ точек.

Почему не DBSCAN:

- Для Haversine матрицы стоимость \(O(N^2)\) и чувствительность к `eps_km` (на проде потребует тонкой калибровки по городам/плотности).

Почему не pure greedy:

- Быстро, но менее предсказуемо по компактности и балансу при изменении распределения точек.
